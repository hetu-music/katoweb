# 河图音乐杯复刻过程文档

## 1. 目标与边界

- 目标页面：`https://lambdaintheshell.com/music-cup/`
- 取证日期：2026-07-22（Asia/Shanghai）
- 本项目实现路由：`/[locale]/music-cup`
- 复刻范围：赛制、交互推进、回退、重置、刷新恢复、赛后展示和分享长图。
- 视觉边界：不复制原站品牌、文案或推荐内容；根据“河图作品勘鉴”现有设计，重做为宣纸、青黛、朱砂的独立视觉。
- 内容边界：原站以周杰伦热门歌曲为样例，本项目固定为河图代表作候选。

## 2. 原站取证

### 2.1 实际走通的完整流程

```text
首页
→ 选择/搜索歌手
→ 加载热门前 48 首
→ 随机分成子、丑、寅、卯、辰、巳、午、未、申、酉、戌、亥十二支，每支 4 首
→ 抽签结果页
→ 12 轮小组赛，每组选择 2 首
→ 24 首直通 + 24 首落选
→ 遗珠复活赛，从 24 首落选歌曲中选择 8 首
→ 组成 32 强
→ 32 强（16 场）
→ 16 强（8 场）
→ 8 强（4 场）
→ 半决赛（2 场）
→ 决赛（1 场）
→ 魁首结果页
→ 绘制问鼎图 / 再开一局
```

关键行为：

- 小组赛最多选 2 首，选满后锁定进入下一组。
- 复活赛最多选 8 首，选满后开启淘汰赛。
- 淘汰赛按上一轮胜者顺序相邻配对；点击一首后短暂停留，再自动进入下一场。
- “返回上一步”撤销最近一次已经提交的操作。冠军页返回后会回到尚未选择胜者的决赛。
- “重开此局”先弹出“此局进度将尽数抹去”的确认框。
- “再开一局”回到首页，不直接沿用歌手重抽。
- 刷新页面后恢复赛事进度。
- 分享弹窗先显示“正在绘制问鼎之路…”，完成后提供长图预览和收图入口。
- 结果页展示魁首、次席和两首四席落选歌曲。

## 3. 河图初选规则

### 3.1 为什么不能称为“热门前 48”

项目曲库包含曲名、专辑、发行日期、创作人员、音频与封面状态等字段，但没有播放量、收藏量或统一热度分。因此无法像原站一样从数据中计算“热门前 48”，也不应伪造热度排名。

本实现采用可复核的编辑候选规则：

1. 固定 48 首跨时期代表作名单，作为编辑候选池；这不是名次排序。
2. 通过稳定的数据库歌曲 ID 匹配，不以简繁体曲名匹配。
3. 优先使用线上/本地曲库返回的真实元数据。
4. 曲库只返回部分数据时，按服务端已有的发行时间倒序，从其他河图歌曲补足。
5. 曲库完全不可用时，使用仓库内置的 48 首最小元数据，保证比赛仍可进行。
6. 每次点击“开始抽签”才对这 48 个 ID 进行 Fisher-Yates 随机洗牌，并切成 12 组。

## 4. 技术设计

### 4.1 页面与数据边界

- Next.js 服务端页面负责读取曲库并选出 48 首候选。
- 客户端组件负责整个比赛过程，赛事状态只保存歌曲 ID，不复制歌曲元数据。
- 封面先使用项目已有的 `getCoverUrl(song)` 规则，失败后改读 `hetu-music.com/og-cover/` 的站点代理和默认封面。
- 曲目档案使用项目的 locale-aware 路由；打开新标签不会中断当前赛事。

### 4.2 状态机

```text
landing → draw → group → wildcard → knockout → result
             └──── 12 组 ────┘      └── 32/16/8/4/2 ──┘
```

状态包含：

- `groups`：12 组固定抽签结果；
- `groupIndex`：当前小组；
- `groupWinners` / `groupLosers`：直通与遗珠集合；
- `pendingSelection`：当前小组或复活赛尚未锁定的选择；
- `wildcards`：8 首复活歌曲；
- `rounds`：每轮的规模、入场顺序与胜者顺序；
- `roundIndex` / `matchIndex`：当前轮次与场次；
- `pendingWinner`：淘汰赛落印动画期间的临时胜者；
- `createdAt`：本场比赛创建时间。

### 4.3 淘汰赛配对

锁定复活名单后，将 24 首小组直通歌曲与 8 首复活歌曲合并并再次洗牌，得到首轮 32 强。每场胜者依照场次顺序写入 `winners`；一轮结束后，按 `winners[0]` 对 `winners[1]`、`winners[2]` 对 `winners[3]` 的方式进入下一轮。因此晋级路径稳定、可回放，也可以准确绘制分享图。

### 4.4 回退与刷新恢复

持久化结构：

```ts
{
  version: 1,
  current: TournamentState,
  history: TournamentState[]
}
```

- Local Storage key：`hetu-music-cup:v1`。
- 每次“已提交操作”前保存完整状态快照：开始抽签、进入小组、锁定小组、锁定复活、每场淘汰赛均可撤销。
- 小组与复活赛的普通勾选会立即持久化，但不会污染撤销历史。
- 淘汰赛点击后的 520ms 落印动画是临时态，持久化时会清空 `pendingWinner`，避免刷新后卡死。
- 恢复时校验 schema 版本、阶段结构和所有引用 ID；候选池变化或存储损坏时安全回到首页。
- “重开此局”和结果页“再开一局”都会清空当前状态与历史。

### 4.5 试听权限差异

原站对候选提供公开试听。本项目现有播放链路由 `/api/navidrome/stream-url`、登录身份、用户 Navidrome 凭据和权益共同控制。实现继续复用现有 `PlayButton`：

- 用户有权益且歌曲标记有音频时显示播放按钮；
- 其他访客仍可打开曲目档案；
- 不创建公开音频地址，也不绕过鉴权。

这是有意保留的产品与安全差异。

### 4.6 分享图

结果页通过客户端 Canvas 生成 `1800 × 2800` PNG：

- 宣纸纹理、青黛墨晕和朱砂冠军摘要；
- 魁首、次席、四席信息；
- 三十二强、十六强、八强、四席争魁、终局问鼎、魁首六列节点；
- 根据每轮胜者关系绘制全部连线；
- 冠军封面依次读取封面 CDN、站点 OG 代理和默认封面，全部失败时退化为文字色块；
- 底部写入站点路径，不伪造二维码；
- 使用 Blob URL 预览并提供 PNG 下载。

## 6. 主要决策结论

- 复刻的是原站的赛事机制与使用节奏，不是对原站视觉和品牌的像素复制。
- 48 首是明确披露的编辑代表作候选，不冒充数据热度榜。
- 随机结果进入状态并持久化，回退不会悄悄重抽。
- 所有需鉴权能力沿用本项目安全边界。
- 赛事可在无 Supabase 的本地环境完整运行，真实曲库可用时自动增强元数据。
